**SQLite VectorDB Specification**

### **Overview**
The SQLite VectorDB is a lightweight, single-file Python library designed to provide vector-based search and retrieval functionality using SQLite as the underlying storage engine. The primary goal is to offer an easy-to-use, self-contained solution that requires no external dependencies like FAISS, Pinecone, or ChromaDB, while still enabling efficient similarity searches. 

This library enables users to store text, images, or any other embedding-based data and query for similar items using cosine similarity directly within SQLite.

### **Key Objectives**
- **Self-contained**: A single Python file that acts as both the interface and implementation.
- **SQLite-based**: Uses SQLite for both metadata storage and efficient vector retrieval.
- **Plug-and-Play**: Can create a new SQLite database or connect to an existing one.
- **Minimal Setup**: No need for separate vector DB infrastructure.
- **Efficient Similarity Search**: Implements cosine similarity to retrieve semantically similar results.
- **Track File Origins**: Ensure stored chunks reference their source files for easy updates and deletion.

### **Library Design**
The VectorDB library will provide the following core functionalities:

#### **1. Database Initialization**
- Can create a new SQLite database file if one doesn’t exist.
- Option to connect to an existing SQLite database.
- A single table will be used to store vectors along with metadata.

#### **2. Data Storage**
- Each entry consists of:
  - A unique ID (autoincremented)
  - Original text or other content reference
  - A vector (stored as a JSON array)
  - Metadata (optional, stored as JSON for flexibility)
  - **File ID or Hash**: Tracks which file the chunk originates from
- Vectors are expected to be floating-point arrays of fixed length.
- Metadata can store additional attributes like timestamps, categories, or labels.

#### **3. Inserting Data**
- Function to insert new text or data into the vector store.
- Embeddings must be computed externally (e.g., using `sentence-transformers` or OpenAI’s embedding API) and passed into the insert function.
- The library does not handle embedding generation—it focuses purely on storage and retrieval.
- **Tracks file origins** to enable efficient file-based updates and deletions.

#### **4. Similarity Search**
- Queries for the `n` most similar vectors using cosine similarity.
- Cosine similarity is computed inside SQLite using a registered Python function.
- Returns results sorted by similarity score.
- Supports filtering by metadata (e.g., “find similar items that were created after a certain date”).

#### **5. Database Management**
- Ability to list all stored vectors.
- Option to delete entries by ID.
- Ability to update metadata while keeping the embedding intact.
- Compact and optimize the database when needed.
- **File-based deletion**: Remove all vectors associated with a given file when needed.

### **Interaction with SQLite**
- Uses SQLite as a persistent backend for vector storage.
- Vectors are stored as JSON arrays inside SQLite.
- A Python function is registered in SQLite to handle similarity calculations.
- Queries are executed using standard SQL statements.
- Performance optimizations will focus on indexing and query structuring.

### **Library Interface**
The library will provide a simple Python interface for usage:
- `connect(db_path: str)`: Connects to an SQLite DB (or creates one if it doesn’t exist).
- `insert(text: str, vector: list, metadata: dict, file_id: str)`: Stores a vector and associated data.
- `search(query_vector: list, top_n: int, filters: dict = None)`: Retrieves the most similar entries.
- `delete(entry_id: int)`: Deletes an entry from the database.
- `delete_by_file(file_id: str)`: Deletes all entries linked to a specific file.
- `list_all()`: Lists all stored vectors and metadata.

### **Performance Considerations**
- Uses SQLite indexing for fast metadata filtering.
- Cosine similarity is computed efficiently using numpy inside SQLite functions.
- JSON storage ensures flexibility without unnecessary complexity.

### **Deployment and Use Cases**
- Works entirely locally—no internet or external services needed.
- Ideal for applications where a simple, portable vector DB is needed without the complexity of FAISS or cloud-hosted options.
- Can be embedded into larger projects that require vector similarity retrieval.
- Supports file-based data management for easy updates and deletions.

### **Conclusion**
This SQLite VectorDB is designed to provide a minimalistic, dependency-free, and highly portable solution for vector search. By leveraging SQLite’s robustness and Python’s simplicity, it allows for efficient and structured storage of embeddings with built-in semantic search capabilities. The entire implementation will be contained within a **single Python file**, making it easy to integrate into any project or use as a standalone tool.
